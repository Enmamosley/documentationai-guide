## Para qué sirve

`/sistema` es el panel admin. Tres bloques en una pantalla:

1. **Checklist Día 0** — qué falta configurar antes de que el equipo empiece a operar.
2. **Usuarios y permisos** — crear cuentas, cambiar rol, desactivar.
3. **Accesos directos** a sub-pantallas (empresa, documentos, catálogos, plantillas).

## Checklist Día 0

Banner arriba de `/sistema`. Lista 5 ítems con estado en vivo (✅/⚠️) y CTA al lugar donde se resuelve cada uno:

<Steps>
  <Step title="Datos fiscales de empresa" icon="building" title-type="p">
    ¿`empresa_config.rfc` está cargado? Si no → CTA "Completar" a `/sistema/empresa`.
  </Step>
  <Step title="Al menos 1 almacén activo" icon="package" title-type="p">
    `count(almacenes WHERE activo=true) > 0`. Si no → CTA "Crear primer almacén" a `/catalogos/almacenes`.
  </Step>
  <Step title="Al menos 1 técnico activo" icon="user" title-type="p">
    `count(tecnicos WHERE activo=true) > 0`. CTA a `/catalogos/tecnicos`.
  </Step>
  <Step title="Catálogo de SKUs poblado" icon="hash" title-type="p">
    Algún SKU registrado (idealmente con factor Telmex). CTA a `/catalogos/skus`.
  </Step>
  <Step title="Al menos 1 usuario operativo (no-admin)" icon="users" title-type="p">
    Algún user con rol ≠ admin. CTA a la sección "Usuarios" abajo en la misma página.
  </Step>
</Steps>

Cuando los 5 están ✅, el banner pasa a verde "Listo para operar" con el siguiente paso (registrar primera remisión).

## Gestión de usuarios

Sección colapsable con la tabla de cuentas + form de creación.

<Steps>
  <Step title="Crear usuario nuevo" icon="user-plus" title-type="p">
    Click en "Crear usuario" → form con email, password (generador random 12 chars sin O/0/l/1), nombre, alias, rol y técnico vinculado (si rol = técnico).

    El action llama `auth.admin.createUser` con `email_confirm=true` (no requiere SMTP — admin entrega credenciales por canal seguro) + `cambiar_rol_usuario` RPC para el rol. Si la asignación falla, rollback automático (delete del user).
  </Step>
  <Step title="Cambiar rol" icon="shield" title-type="p">
    Botón "Cambiar rol" por fila → inline select de rol + (si elige técnico) dropdown del catálogo. Llama `cambiar_rol_usuario` RPC.
  </Step>
  <Step title="Desactivar / Reactivar" icon="user-x" title-type="p">
    `ban_duration: "876000h"` (~100 años) cuando desactiva. No borra el user para preservar audit trail (OS, EMU, consumo histórico que referencien su `tecnico_id`). `"none"` para reactivar.

    **No podés desactivarte a vos mismo** — server-side guard.
  </Step>
</Steps>

<Callout kind="alert">
  **Setup requerido**: para crear o desactivar usuarios desde la UI, necesitás configurar `SUPABASE_SERVICE_ROLE_KEY` en env vars (ver `.env.example`). Sin eso, la sección muestra un banner amarillo y los botones se ocultan — sólo podés cambiar el rol de cuentas que ya existan vía signup.
</Callout>

<Callout kind="alert">
  Después del bug 4 (commit `6436986`), **un usuario no puede escalar su propio rol**. La RLS policy `Users update own profile (no rol/tecnico_id)` con `WITH CHECK` lo bloquea. El trigger `handle_new_user` ignora `raw_user_meta_data.rol` y hardcodea `'tecnico'` al signup.
</Callout>

## `/sistema/empresa` · Datos fiscales y branding

Editor único de `empresa_config` (fila singleton `id=1`). Campos:

| Campo | Para qué se usa | Notas |
|---|---|---|
| `razon_social` | Nombre legal completo en CFDIs, PDFs, "si encuentra esta credencial" en credencial | Va tal cual al SAT |
| `nombre_comercial` | Brand corto en headers de R20, vale, ficha, credencial. Si está null, deriva de las primeras 2 palabras de `razon_social` | Ej. "WCG SERVICIOS" |
| `rfc` | Header PDFs, emisor CFDI, credencial | Validado uppercase en cliente |
| `regimen_fiscal` | CFDI `RegimenFiscal` del emisor | Catálogo SAT `c_RegimenFiscal` |
| `tasa_iva` | **Factor IVA** para `/facturas/nueva` (subtotal × tasa). Guardado como fracción 0–1, editado como porcentaje 0–100 | 0.16 nacional · 0.08 frontera norte · 0 exento |
| `domicilio_fiscal` | Dirección legal en CFDI + credencial reverso | Una línea |
| `cp_lugar_expedicion` | Atributo `LugarExpedicion` del CFDI | 5 dígitos |
| `repse_folio`, `ar_aviso_registro` | Documentación REPSE para subcontratación | Visible en credencial / ficha si se quiere |
| `certificado_csd_serial` + `_vencimiento` | Datos del CSD para emisión de CFDI (Fase 3+) | Banner amarillo si vence en <30 días, rojo si ya venció |

<Callout kind="info">
  Después del hallazgo de hardcodes (commit `e17c884`), **los 4 PDFs**
  (R20, vale FR-PM-12, ficha técnica, credencial) leen estos campos
  dinámicamente. Cambiar la razón social o el RFC acá se ve reflejado
  en el siguiente render del PDF, sin redeploy.
</Callout>

## Permisos

| Acción | Roles |
|---|---|
| Ver `/sistema` | **admin** |
| Crear usuario | admin (requiere SERVICE_ROLE_KEY) |
| Cambiar rol | admin |
| Desactivar / reactivar | admin (requiere SERVICE_ROLE_KEY) |
| Editar empresa | admin |
| Subir documentos | todos excepto técnico |

## Server Actions

```
src/lib/actions/users.ts
  cambiarRolUsuarioAction(userId, rol, tecnicoId)
    // Admin-only. RPC cambiar_rol_usuario que valida en SQL.
    // Bloquea auto-degradación del último admin.
  listUsersForAdminAction()
    // Admin-only. RPC list_users_for_admin con JOIN auth.users (email + banned_until).
  crearUsuarioAction(input: CrearUsuarioInput)
    // Requiere SERVICE_ROLE_KEY. createUser + cambiar_rol_usuario con rollback.
  desactivarUsuarioAction(userId)
  reactivarUsuarioAction(userId)
    // ban_duration: "876000h" / "none".

src/lib/actions/empresa.ts
  actualizarEmpresaAction(patch)

src/lib/actions/documentos.ts
  subirDocumentoAction(formData)
  getDocumentoDownloadUrlAction(documentoId)
  eliminarDocumentoAction(documentoId)
```
