## Qué es una OS

La **Orden de Servicio** es la unidad de trabajo. Cada OS representa un servicio que Telmex pide vía SIATEL (instalación nueva, migración, reparación) y que WCG ejecuta en campo.

<Callout kind="info">
  WCG **no** tiene acceso al SIATEL de Telmex. La OS se captura en la app **después** de hacer el servicio, con el folio SIATEL como referencia.
</Callout>

## Calendario (`/os/calendario`)

Pantalla nueva con dos modos de visualización, útil para que el supervisor planifique carga semanal y detecte huecos/sobreasignación.

<Steps>
  <Step title="Vista Calendario" icon="calendar" title-type="p">
    Grilla 6×7 estándar (lunes a domingo). Cada celda lista las OS del día con chip de color por estado (Capturada / Liquidada / Facturada / Pagada / Cancelada). Click en una OS lleva al detalle.
  </Step>
  <Step title="Vista Heatmap" icon="grid" title-type="p">
    Matriz técnicos × días del mes con intensidad por cantidad de OS. Identifica de un vistazo qué técnico está saturado o sub-asignado en una semana específica.
  </Step>
</Steps>

Controles compartidos:

- **Switch fecha**: `fecha` (captura) o `fechaCierre` (liquidación). Útil para ver "qué se planificó" vs "qué se cerró" en el mes.
- **Filtros**: estado, técnico, tipo de servicio. Aplican en cliente sin re-fetch (la cantidad de OS al año es chica — cientos — no vale paginar).
- **Navegación**: ‹ Mes anterior · Mes siguiente › · botón "Hoy". Sin recargar el server.
- **Export CSV** del periodo visible con los filtros aplicados.

RLS aplica igual: técnico ve sólo sus OS, así que su calendario es su agenda personal.

## Captura en `/os/nueva` (4 steps)

<Steps>
  <Step title="0. Datos del servicio" icon="tag" title-type="p">
    Tipo de servicio (Alta nuevo, etc.), número de altas, folio SIATEL (obligatorio), subtipo TELMEX (`alta_ftth` | `migracion_ftth` | `migracion_ftth_vieja` | `no_aplica`), modelo de ONT.
  </Step>
  <Step title="1. Cliente y domicilio" icon="user" title-type="p">
    Nombre, teléfono, tipo cliente, calle + número (interior si aplica),
    CP, colonia, alcaldía / municipio, estado (dropdown con los 32
    estados de México — antes sólo CDMX/Edomex/Morelos/Hidalgo), referencia.
  </Step>
  <Step title="2. Asignación y material" icon="package" title-type="p">
    Técnico asignado (forzado a sí mismo si rol=tecnico). Default para
    admin/ops: primer técnico activo del catálogo (antes era `T-014`
    hardcoded). Alfanuméricos de las ONTs a entregar. Vista previa de
    material sugerido `factor × altas` — read-only, sólo informa cuánto
    se sembrará en `os_materiales`.
  </Step>
  <Step title="3. Revisión" icon="check" title-type="p">
    Resumen antes de capturar. Click "Capturar OS" llama a `crear_os_atomica`: folio + INSERT os + seed `os_materiales` con `cantidad_solicitada = ceil(factor × altas)` y `cantidad_consumida = NULL`. **No descuenta mochila aún** — eso pasa al liquidar.
  </Step>
</Steps>

<Callout kind="info">
  Los campos del R20 (potencia óptica, distrito, terminal, upsell Infinitum) NO se capturan en este wizard — se editan **después** desde la tab "Mediciones" en `/os/[id]` via `editarMedicionesOSAction`. Es post-creación porque el técnico suele tener el R20 físico en mano al volver a la oficina.
</Callout>

## Flujo de estados

```
Capturada → Liquidada → Facturada → Pagada
                ↓
            Cancelada   (estado existe, sin flujo expuesto)
```

- **Capturada**: existe en BD con `os_materiales` sembrados; consumo NULL.
- **Liquidada** (`liquidarOSAction` → RPC `liquidar_os_atomica`): para cada partida con consumo NULL, setea consumida=entregada y descuenta mochila atómicamente (Bug 3). La OS pasa a aparecer en el F200.
- **Facturada**: se incluye en un CFDI emitido a CARSO (set vía `actualizarCobranzaAction`).
- **Pagada**: CARSO depositó. Trigger `emu_avanzar_cobrado` avanza filas EMU asociadas a `cobrado`.

<Callout kind="info">
  **Cancelar OS** (B2 · commit `bb19724`): el botón "Cancelar OS" aparece en el detalle para admin/operaciones cuando la OS está Capturada o Liquidada. Pide motivo (mín 3 chars). Si estaba Liquidada, la RPC `cancelar_os_atomica` revierte el consumo de cada partida devolviendo el material a la mochila del técnico atómicamente y limpia `cantidad_consumida = 0`. OS Facturada/Pagada NO se pueden cancelar desde acá — hay que cancelar el CFDI primero.
</Callout>

## Permisos

| Acción | Roles |
|---|---|
| Capturar OS nueva | admin, operaciones, **técnico** (forzado a sí mismo) |
| Liquidar OS | admin, operaciones, técnico (sólo las suyas via RLS) |
| Editar mediciones R20 (`editarMedicionesOSAction`) | admin, operaciones, técnico (sólo las suyas) |
| Editar notas técnico | admin, técnico (sólo las suyas) |
| Agregar partida manual (`agregarPartidaOSAction`) | admin, operaciones, almacenista, técnico |
| Cancelar OS (`cancelarOSAction`) | admin, operaciones (técnico NO — supervisión requerida) |

<Callout kind="alert">
  El rol **técnico** es auto-servicio. Ve sólo SUS OS, captura con su mochila, liquida lo suyo. RLS filtra `auth.uid()` contra `tecnico.user_id`.
</Callout>

## Server Actions

```
src/lib/actions/os.ts
  createOSAction(input)              # → RPC crear_os_atomica (folio + INSERT atómico)
  liquidarOSAction(osId)             # → RPC liquidar_os_atomica (auto-consumo + mochila−)
  agregarPartidaOSAction(osId, sku, cantidad)
                                     # → RPC agregar_partida_os (INSERT…ON CONFLICT suma)
  actualizarEntregadaAction(osId, sku, qty)
                                     # almacenista actualiza ENTREGADA
  reportarConsumoAction(osId, sku, qty)
                                     # técnico reporta CONSUMIDA (delta vs previo → mochila)
  editarMedicionesOSAction(osId, patch)
                                     # campos R20 + RTF + upsell Infinitum
  editarNotasTecnicoAction(osId, notas)
                                     # texto libre del técnico
  cancelarOSAction(osId, motivo)
                                     # → RPC cancelar_os_atomica
                                     # Sólo Capturada/Liquidada. Si Liquidada,
                                     # devuelve material a la mochila del técnico
                                     # atómicamente (Bug del backlog B2).
```

## R20 PDF

`GET /api/os/[id]/r20` genera el PDF imprimible del formato R20 con los
datos capturados. Útil para entregar al cliente o archivar físicamente.

El header del PDF (marca, razón social, RFC) viene de `empresa_config`
vía helper `empresaBrand()` — antes estaba hardcoded. Mismo patrón en
los otros 3 PDFs del sistema:

| Ruta | PDF | Header dinámico |
|---|---|---|
| `/api/os/[id]/r20` | Reporte R20 (formato Telmex FR-PE-58) | ✅ |
| `/api/remisiones/[numero]/vale` | Vale FR-PM-12 (salida de material) | ✅ |
| `/api/tecnicos/[id]/ficha` | Ficha técnica REPSE | ✅ |
| `/api/tecnicos/[id]/credencial` | Credencial PVC frente + reverso | ✅ (incluye sección "Si encuentra esta credencial" con razón social + RFC + domicilio) |
