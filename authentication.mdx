## Modelo de auth

WCG App **no expone una API pública** con tokens. Toda la autenticación pasa por la **sesión de Supabase** establecida en `/login`, que se materializa como **cookies HTTP-only** firmadas. Los endpoints en `/api/*` validan esa sesión leyendo las cookies con `getSupabaseServer()`.

<Callout kind="info" collapsed="false">
  No hay `Authorization: Bearer …`. No hay API keys de usuario. La única forma de tocar `/api/*` es estar logueado en el navegador.
</Callout>

## Flujo

<Steps>
  <Step title="Login" icon="lock" title-type="p">
    POST a `/login` con email + password vía la action `loginAction`. Supabase responde con `Set-Cookie` de la sesión.
  </Step>
  <Step title="Middleware refresca tokens" icon="refresh-cw" title-type="p">
    `middleware.ts` corre en cada request. Lee cookies, refresca el JWT si está por expirar, y lo escribe de vuelta.
  </Step>
  <Step title="Endpoint lee sesión" icon="server" title-type="p">
    El handler en `/api/*` llama `await fetchSessionUser()` (que internamente usa `getSupabaseServer()`). Si no hay user → 401. Si el rol no aplica → 403.
  </Step>
  <Step title="Postgres aplica RLS" icon="shield" title-type="p">
    Cualquier query del handler viaja con el JWT del usuario. RLS evalúa contra `current_user_role()`.
  </Step>
</Steps>

## Ejemplo: descarga F200

```bash
# Asumiendo que la cookie de sesión ya está en ~/.cookies.txt:
curl https://wcg-app-v2.vercel.app/api/f200?periodo=2026-05-W19 \
  -b ~/.cookies.txt \
  -o F200_2026-05-W19.xlsx
```

Si no hay cookie válida:

```
HTTP/2 401
content-type: text/plain

No autenticado
```

Si la cookie es válida pero el rol no tiene permiso:

```
HTTP/2 403
content-type: text/plain

Sin permiso para generar F200
```

## Rutas públicas (sin auth)

| Path | Notas |
|---|---|
| `/login` | Página de login |
| `/auth/callback` | Callback OAuth de Supabase |

Todo lo demás (incluyendo `/api/*`) requiere sesión. El matcher del middleware está en:

```typescript
// middleware.ts
export const config = {
  matcher: ['/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)']
};
```

## Validación por rol en endpoints

Cada handler implementa su gate. Ejemplo de `/api/f200`:

```typescript
const session = await fetchSessionUser();
if (!session) return new NextResponse('No autenticado', { status: 401 });

const rol = session.profile.rol;
if (rol !== 'admin' && rol !== 'operaciones' && rol !== 'contable') {
  return new NextResponse('Sin permiso para generar F200', { status: 403 });
}
```

Esto duplica intencionalmente la gate del cliente (`canView.f200(rol)`) para defensa en profundidad.

## ¿Por qué no JWT API?

WCG App está pensada como herramienta interna. Si en el futuro hay que exponer integraciones (CARSO, Telmex), se agregará un esquema separado con:

- API keys por integración (no por usuario)
- Scopes mínimos
- Rate limiting
- Logging dedicado

Por ahora ese esquema **no existe**.
