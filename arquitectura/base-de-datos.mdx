## RLS (Row-Level Security)

<Callout kind="alert" collapsed="false">
  **RLS está ACTIVO en todas las tablas.** Sin sesión válida, todas las queries fallan con `42501`. No se confía en el cliente — la seguridad vive en Postgres.
</Callout>

### Lectura

<Callout kind="alert" collapsed="false">
  **Cambio importante (commit `d141f9f`, parche D).** Antes de este commit, casi todas las tablas tenían una policy uniforme `USING (true)`, lo que permitía que un técnico leyera expedientes ajenos, facturas, etc. vía `supabase-client` desde devtools. Ahora las SELECT policies están endurecidas por rol y ownership.
</Callout>

| Tabla | Quién puede SELECT |
|---|---|
| `tecnicos_expediente` | admin/operaciones/contable todos · técnico **sólo el suyo** |
| `facturas_emitidas` | admin/contable |
| `documentos` | operativos todos · técnico **sólo los vinculados a él o a una OS suya** |
| `os` · `onts` · `emu_rows` · `inventory` | admin/operaciones/almacenista (+contable según) · técnico **filtrado por su `tecnico_id`** |
| `os_materiales` · `*_items` | heredado del parent table vía subquery |
| `remisiones` · `devoluciones` · `traspasos` · `salidas_tecnico` | operativos · técnico ve las suyas cuando aplica |
| `tecnicos_herramienta` · `tecnicos_vehiculo` | admin/operaciones/contable · técnico **la suya** |
| `user_profiles` | admin todos · resto **sólo su propio profile** |
| `programas_obra` · `programa_dias` | admin/operaciones |
| Catálogos puros (`skus`, `almacenes`, `tecnicos`, `copes`, `tipos_tarea`, `empresas_empleadoras`, `emu_conceptos`, `unidades_medida`, `clientes`, `empresa_config`) | `to authenticated using (true)` — son data de referencia sin la cual nada funciona |

Para ver "asignado a X" desde un contexto distinto a `/sistema`, usar el RPC `list_users_for_admin` en vez de leer `user_profiles` directo.

### Mutaciones

Cada tabla tiene policies por rol via `current_user_role()`:

| Tabla | INSERT | UPDATE |
|---|---|---|
| `os` | admin · operaciones · tecnico (suya) | admin · operaciones · tecnico (suya) |
| `onts` | admin · almacenista | admin · operaciones · almacenista · tecnico (asignada — `liquidarONTAction` valida ownership de la OS, Bug 11) |
| `facturas_emitidas` | admin · contable | admin · contable |
| `inventory` | admin · almacenista | admin · almacenista (vía triggers atómicos) |
| `remisiones` | admin · almacenista · operaciones | admin · almacenista |
| `*_items` (remision/devolucion/traspaso/salida_tecnico) | mismo rol que el parent · UNIQUE `(parent_id, sku)` (Bug 10) | igual |
| `emu_rows` | trigger SQL (auto al liquidar OS) | admin · contable |
| `user_profiles` | admin | admin |

### Helper SQL

```sql
CREATE FUNCTION current_user_role() RETURNS user_role AS $$
  SELECT rol FROM user_profiles WHERE id = auth.uid()
$$ LANGUAGE sql STABLE;
```

## Triggers atómicos

<Steps>
  <Step title="liquidar_os → consume mochila" icon="package-minus" title-type="p">
    Cuando `os.estado` pasa a `Liquidada`, un trigger BEFORE UPDATE descuenta de `inventory` (mochila del técnico) las cantidades de `os_partidas`. Si no hay stock, falla con `23514` y la mutación se aborta.
  </Step>

  <Step title="confirmar_remision → incrementa almacén" icon="package-plus" title-type="p">
    Al pasar `remisiones.estado` a `Confirmada`, las partidas suben el `disponible` del almacén destino. Atómico via row-level locking.
  </Step>

  <Step title="factura_pagada → EMU cobrado" icon="check-circle" title-type="p">
    Cuando `facturas.estado` pasa a `Pagada`, todas las `emu_rows` con `factura_id = NEW.id` pasan a `Cobrado` y se sella `fecha_cobro`.
  </Step>

  <Step title="prevent_role_escalation" icon="shield" title-type="p">
    Trigger en `user_profiles` BEFORE UPDATE: si `NEW.rol != OLD.rol` y `auth.uid() != admin`, se aborta. (Bug 4 cerrado en `6436986`.)
  </Step>
</Steps>
