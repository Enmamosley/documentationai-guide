## Dos capas, dos propósitos

```
┌─────────────────────────────────────────┐
│  Cliente: canView · can                 │ ← UX (esconder botones/rutas)
│  src/lib/permissions.ts                 │
├─────────────────────────────────────────┤
│  Servidor: RLS policies en Postgres     │ ← Seguridad real
│  current_user_role() = NEW.rol          │
└─────────────────────────────────────────┘
```

<Callout kind="alert" collapsed="false">
  Las funciones `can.X(rol)` y `canView.X(rol)` en el cliente **no son la frontera de seguridad** — sirven para UX. La frontera es RLS en Postgres. Si un cliente "forzara" una mutación que su rol no permite, Postgres responde `42501`.
</Callout>

## Cómo funciona

<Steps>
  <Step title="Middleware redirige sin sesión" icon="lock" title-type="p">
    `middleware.ts` corre en cada request. Si no hay user, redirige a `/login?from=<path>`. Rutas públicas: `/login`, `/auth/callback`.
  </Step>

  <Step title="Server Component filtra rutas" icon="filter" title-type="p">
    Cada `page.tsx` consulta el rol con `fetchSessionUser()` y llama `canView.X(rol)`. Si false → `redirect("/")`.
  </Step>

  <Step title="Sidebar filtra items" icon="list" title-type="p">
    `SideNav.tsx` recibe el rol y oculta los items para los que `canView.X` es false. Eso evita confusión con secciones que no le tocan.
  </Step>

  <Step title="Server Action ejecuta mutación" icon="zap" title-type="p">
    El usuario interactúa con un botón → Server Action → `getSupabaseServer()` → JWT viaja a Postgres → RLS evalúa contra `current_user_role()`.
  </Step>

  <Step title="Postgres es la palabra final" icon="database" title-type="p">
    Si la policy no permite, retorna `42501` (RLS violation). El UI muestra el error sin haber tocado datos.
  </Step>
</Steps>

## Por qué el técnico ve sólo lo suyo

Para el rol `tecnico`, las RLS policies filtran por su `tecnico_id` resuelto desde `user_profiles`. La función helper es **`current_user_tecnico_id()`** (NO `tecnicos.user_id` — la tabla `tecnicos` no tiene esa columna; el vínculo vive en `user_profiles.tecnico_id`).

```sql
-- Helper (existe en DB)
CREATE FUNCTION current_user_tecnico_id() RETURNS text AS $$
  SELECT tecnico_id FROM user_profiles WHERE id = auth.uid()
$$ LANGUAGE sql STABLE;

-- Patrón típico en SELECT policies post-D
CREATE POLICY "Tecnico lee sus OS" ON os
  FOR SELECT TO authenticated
  USING (current_user_role() = 'tecnico'
         AND tecnico_id = current_user_tecnico_id());
```

Este patrón se replica en `onts`, `emu_rows`, `inventory`, `tecnicos_expediente`, `tecnicos_herramienta`, `tecnicos_vehiculo`, `documentos`, `salidas_tecnico`. El técnico no necesita filtros en la UI — Postgres ya devuelve sólo sus filas.

<Callout kind="alert">
  **Antes** del parche D, casi todas las tablas tenían `USING (true)` en SELECT. Un técnico podía leer expedientes ajenos, facturas y documentos via `supabase-client` desde devtools. Ahora las SELECT policies son por rol/ownership. Ver detalle en [Base de datos · RLS](/arquitectura/base-de-datos#rls-row-level-security).
</Callout>

## Bug 4: escalation de rol cerrado

<Callout kind="alert">
  Antes del commit `6436986`, un usuario podía cambiar su propio `rol` vía la action de update perfil. Ahora:
  - El trigger `handle_new_user` (sobre `auth.users`) ignora `raw_user_meta_data.rol` y hardcodea `'tecnico'` como rol inicial.
  - El **self-update** de `user_profiles` está cortado por una RLS policy `Users update own profile (no rol/tecnico_id)` con `WITH CHECK` que exige que `rol` y `tecnico_id` queden iguales a los actuales — un `update({rol:'admin'})` falla con `42501`.
  - Sólo admin cambia roles vía RPC `cambiar_rol_usuario(p_user_id, p_rol, p_tecnico_id)` expuesta a través de la UI en `/sistema`.
</Callout>
