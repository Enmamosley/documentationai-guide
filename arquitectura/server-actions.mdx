## Por qué Server Actions

Toda mutación pasa por una Server Action (`'use server'`) en `src/lib/actions/`. Razones:

1. **RLS evalúa con el JWT real del usuario** — el cliente Supabase del servidor lee cookies con `getSupabaseServer()` y la query llega autenticada a Postgres.
2. **Sin tokens secretos en el cliente** — sólo viaja la publishable key (anon).
3. **Validación en un solo lugar** — Zod no es necesario porque Postgres + RLS rechazan inputs inválidos.

<Callout kind="alert">
  **Nunca mutar desde Client Components** con `fetch('/api/...', { method: 'POST' })`. Usa Server Actions. Las únicas Route Handlers son para **descargas** (PDFs, XLSX, CSVs).
</Callout>

## Patrón estándar

```typescript
'use server';

import { getSupabaseServer } from '@/lib/supabase-server';
import { revalidatePath } from 'next/cache';

export async function liquidarOSAction(osId: string) {
  const supabase = await getSupabaseServer();

  // RLS hace el trabajo pesado: si el usuario no puede, falla acá.
  const { error } = await supabase
    .from('os')
    .update({ estado: 'Liquidada' })
    .eq('id', osId);

  if (error) {
    // 42501 → RLS violation
    // 23514 → trigger atómico (stock insuficiente)
    throw new Error(traducirError(error));
  }

  revalidatePath(`/os/${osId}`);
  revalidatePath('/os');
}
```

## Errores frecuentes

| Código PG | Significado | Causa típica |
|---|---|---|
| `42501` | RLS violation | El rol no tiene policy de INSERT/UPDATE para esa tabla |
| `23514` | Check constraint violation | Trigger atómico de stock detectó cantidad < 0 |
| `23505` | Unique violation | Folio duplicado, alfanumérico repetido |
| `23503` | FK violation | Referencia a un tecnico/almacen que no existe |

## Revalidación

Después de cada action:

```typescript
revalidatePath('/ruta/afectada');
```

Esto invalida el cache del Server Component y el siguiente render hace fetch fresco a Supabase.
